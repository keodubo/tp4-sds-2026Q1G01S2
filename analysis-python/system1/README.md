# System 1 Analysis

Postprocessing for TP4 System 1 lives here and stays independent from the Java simulation motor.

## Enunciado Coverage

| Enunciado item | Covered by |
|---|---|
| `1.1` | Java raw trajectory generation only. The Java motor writes `outputs/system1.csv`. |
| `1.2` | This analysis script computes ECM and analytical-vs-numerical comparison figures from `outputs/system1.csv`. |
| `1.3` | This analysis script generates the log-log ECM-vs-`dt` figure and ranking summary from the ECM data. |

The Java motor alone is enough for `1.1`, but complete System 1 evidence also requires the analysis artifacts for `1.2` and `1.3`.

## Inputs

- `outputs/system1.csv`: raw trajectory CSV generated by the Java engine.
- Metadata lines beginning with `#` provide the physical parameters used to reconstruct the analytical solution.

## Outputs

- `outputs/system1_ecm.csv`: one ECM row per `(method, dt)`.
- `outputs/system1_figures/system1_position_dt_0.001.png`: analytical vs numerical position comparison for `dt=0.001`.
- `outputs/system1_figures/system1_position_dt_0.001_zoom.png`: zoomed analytical vs numerical position comparison for `dt=0.001`.
- `outputs/system1_figures/ecm_vs_dt.png`: log-log ECM vs `dt` comparison for all System 1 methods.
- `outputs/system1_summary.md`: compact method ranking at the smallest generated `dt`.
- `outputs/system1_outputs_manifest.csv`: maps generated artifacts to TP4 incisos.

## Commands

Generate or refresh the raw Java trajectory first:

```bash
cd SdS_TP4_2026Q1G01CS2_Codigo
mvn exec:java -Dexec.args="system1 --output ../outputs/system1.csv"
```

Then compute the analytical comparison data:

```bash
python3 analysis-python/system1/analyze_system1.py \
  --input outputs/system1.csv \
  --ecm-output outputs/system1_ecm.csv \
  --figures-dir outputs/system1_figures \
  --figure-dt 0.001
```

The `--figure-dt` values must already exist in the Java trajectory CSV. Rerun the Java command with the desired `--dt` values before requesting additional figures.

Generate the full System 1 item `1.3` ECM-vs-`dt` study and method ranking:

```bash
python3 analysis-python/system1/analyze_system1.py \
  --input outputs/system1.csv \
  --ecm-output outputs/system1_ecm.csv \
  --figures-dir outputs/system1_figures \
  --figure-dt 0.001 \
  --summary-output outputs/system1_summary.md
```

Run the lightweight analysis tests:

```bash
python3 -m unittest discover -s analysis-python/system1 -p 'test_*.py'
```

## Output Manifest Convention

`outputs/system1_outputs_manifest.csv` is a generated handoff index with one row per relevant artifact:

```csv
system,inciso,artifact_type,path,description
```

System 1 uses these mappings:

- `outputs/system1.csv`: `system=1`, `inciso=1.1`, `artifact_type=raw-trajectory`.
- `outputs/system1_ecm.csv`: `system=1`, `inciso=1.2`, `artifact_type=analysis-data`.
- `outputs/system1_figures/system1_position_dt_*.png`: `system=1`, `inciso=1.2`, `artifact_type=figure`.
- `outputs/system1_figures/ecm_vs_dt.png`: `system=1`, `inciso=1.3`, `artifact_type=figure`.
- `outputs/system1_summary.md`: `system=1`, `inciso=1.3`, `artifact_type=summary`.

Future System 2 postprocessing should use the same columns and declare the matching TP4 inciso for each output, including figures and animations.

## Scope

This phase computes analytical position and velocity, position ECM, analytical-vs-numerical figures, ECM-vs-`dt` analysis, and method ranking from existing Java output.

Generated outputs, figures, and this Python analysis code are not part of the final Java code ZIP. The campus code ZIP should contain only the final Java simulation motor source under `SdS_TP4_2026Q1G01CS2_Codigo/`.
